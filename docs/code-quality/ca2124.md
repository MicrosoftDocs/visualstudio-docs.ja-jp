# <a name="ca2124-wrap-vulnerable-finally-clauses-in-outer-try"></a>CA2124:脆弱性のある finally 句を外側の try でラップします

|||
|-|-|
|TypeName|WrapVulnerableFinallyClausesInOuterTry|
|CheckId|CA2124|
|カテゴリ|Microsoft.Security|
|互換性に影響する変更点|なし|

## <a name="cause"></a>原因
.NET Framework のバージョン1.0 および1.1 では、パブリックメソッドまたはプロテクトメソッドに @no__t 0 @ no__t @ no__t @ @ ブロックが含まれています。 @No__t-0 ブロックがセキュリティ状態をリセットし、`finally` ブロックで囲まれていません。

## <a name="rule-description"></a>規則の説明
このルールでは、呼び出し履歴に存在する悪意のある例外フィルターに対して脆弱である可能性のある、.NET Framework のバージョン1.0 および1.1 を対象とするコード内の @no__t 0 @ no__t-1 @ no__t ブロックを検索します。 偽装などの機密性の高い操作が try ブロックで発生し、例外がスローされた場合、フィルターは @no__t 0 ブロックの前に実行できます。 偽装の例では、これは、フィルターが権限を借用したユーザーとして実行されることを意味します。 現在、フィルターは Visual Basic のみで実装できます。

> [!NOTE]
> .NET Framework のバージョン2.0 以降では、例外ブロックを含むメソッド内でリセットが直接行われた場合、ランタイムは、悪意のある例外フィルターから @no__t 0 @ no__t-1 @ no__t @ no__t を自動的に保護します。

## <a name="how-to-fix-violations"></a>違反の修正方法
ラップされていない `try` @ no__t-1 @ no__t を外側の try ブロックに配置します。 次の2番目の例を参照してください。 これにより、フィルターコードの前に @no__t 0 が強制的に実行されます。

## <a name="when-to-suppress-warnings"></a>警告を非表示にする場合
この規則による警告は抑制しないでください。

## <a name="pseudo-code-example"></a>擬似コードの例

### <a name="description"></a>説明

次の擬似コードでは、このルールにより検出されたパターンを示しています。

```csharp
try {
   // Do some work.
   Impersonator imp = new Impersonator("John Doe");
   imp.AddToCreditCardBalance(100);
}
finally {
   // Reset security state.
   imp.Revert();
}
```

次の擬似コードは、コードを保護し、この規則を満たすために使用できるパターンを示しています。

```csharp
try {
     try {
        // Do some work.
     }
     finally {
        // Reset security state.
     }
}
catch()
{
    throw;
}
```
