# <a name="general-debugging-options"></a>全般的なデバッグ オプション
Visual Studio デバッガーのオプションを設定するには、次のように選択します**ツール** > **オプション**、および **デバッグ**を選択するか、ボックスの横に選択を解除、  **。一般的な**オプション。 すべての既定設定に戻すことができます**ツール** > **インポートおよびエクスポート設定** > **すべての設定をリセット**します。 設定のサブセットをリセットすると設定を保存、**インポートおよびエクスポート設定ウィザード**テストに必要な変更を行う前にし、設定をインポート保存した後。
  
次を設定する**全般**オプション。

**すべてのブレークポイントを削除する前に確認**: 完了する前に確認が必要です、**すべてのブレークポイントの削除**コマンド。  
  
**1 つのプロセスがブレークするときに、すべてのプロセスをブレーク**。 同時に、デバッガーがアタッチされている、中断が発生した場合、すべてのプロセスを中断します。  
  
**例外が AppDomain またはマネージド/ネイティブ境界を越える場合にブレーク**: マネージドまたは混合モード デバッグ、共通言語ランタイムがアプリケーション ドメインの境界またはマネージド/ネイティブ境界を越える例外をキャッチできる場合、次条件が true:  
  
1. COM 相互運用機能とマネージ コードを使用して、ネイティブ コードがマネージ コードを呼び出すときに例外をスローします。 参照してください[COM 相互運用機能の概要](/dotnet/articles/visual-basic/programming-guide/com-interop/introduction-to-com-interop)します。  
  
2. アプリケーション ドメイン 1 で実行されるマネージ コードは、アプリケーション ドメイン 2、マネージ コードを呼び出し、アプリケーション ドメイン 2 のコードが例外をスローします。 参照してください[アプリケーション ドメインを使用したプログラミング](/dotnet/articles/framework/app-domains/index)します。  

3. コードは、リフレクションを使用して関数を呼び出し、その関数が例外をスローします。 参照してください[リフレクション](/dotnet/framework/reflection-and-codedom/reflection)します。  
  
2 および 3 の条件で例外が場合がありますでマネージ コードによってキャッチ`mscorlib`共通言語ランタイムではなく、します。 このオプションを選択しても、`mscorlib` でキャッチされる例外の処理は中断されません。  
  
**アドレス レベルのデバッグを有効にする**: アドレス レベルのデバッグの詳細設定を有効に (、**逆アセンブリ**ウィンドウで、**登録**ウィンドウ、およびアドレス ブレークポイント)。  
  
- **ソースがないかどうかは逆アセンブリの表示**: 自動的に表示されます、**逆アセンブル**ソースが利用しないコードをデバッグするときにウィンドウ。  
  
**ブレークポイントのフィルターを有効にする**: 特定のプロセス、スレッド、またはコンピューターのみが影響するために、ブレークポイントのフィルターを設定することができます。  
 
**新しい例外ヘルパーを使用して、**: 例外処理アシスタントを置換する例外ヘルパー (Visual Studio 2017) を使用します。
  
> [!NOTE]
> マネージ コードは、このオプションが呼び出された**例外処理アシスタントを有効にする**します。 
  
**マイ コードのみを有効にする**: デバッガーが表示され、システムのコードと最適化されたデバッグ シンボルがないその他のコードを無視してユーザー コード (「マイ コード」) のみにステップします。

- **起動 (マネージドのみ) にユーザー コードが見つからない場合は、警告**: マイ コードのみを有効になっているとデバッグが始まり、このオプション警告を表示するかどうかは、ユーザー コード (「マイ コード」) はありません。 

**.NET Framework を有効にするソースのステッピング**: により、デバッガーは .NET Framework のソースにステップ インできます。 このオプションを自動的に有効にするには、マイ コードのみを無効にします。 .NET framework シンボルは、キャッシュの場所にダウンロードされます。 キャッシュの場所を変更、**オプション**ダイアログ ボックスで、**デバッグ**カテゴリ、**シンボル**ページ。  
  
**プロパティおよび演算子 (マネージドのみ) をステップ オーバー**: デバッガーがプロパティとマネージ コードでの演算子にステップ インするを防ぎます。  
  
**プロパティの評価とその他の暗黙的な関数呼び出しを有効にする**: 変数ウィンドウに呼び出しを有効の自動評価のプロパティと暗黙的な関数と**クイック ウォッチ**  ダイアログ ボックス。  
  
- **変数ウィンドウ内のオブジェクトに対して文字列変換関数を呼び出す (C#および JavaScript のみ)**: 変数ウィンドウ内のオブジェクトを評価するときに、文字列の暗黙的な変換の呼び出しを実行します。 結果は、型名の代わりに、文字列として表示されます。 C# コードのデバッグ時にのみ適用されます。 この設定は、DebuggerDisplay 属性によってオーバーライドできます (を参照してください[DebuggerDisplay 属性を使用して](../debugger/using-the-debuggerdisplay-attribute.md))。  
  
**ソース サーバー サポートを有効にする**: SrcSrv を実装するソース サーバーからソース ファイルを取得する Visual Studio デバッガーに指示 (`srcsrv.dll`) プロトコル。 このプロトコルを実装する 2 つのソース サーバーは Team Foundation Server と Debugging Tools for Windows です。 SrcSrv のセットアップの詳細については、次を参照してください。、 [SrcSrv](/windows-hardware/drivers/debugger/srcsrv)ドキュメント。 さらを参照してください[シンボル (.pdb) ファイルとソース ファイルを指定](../debugger/specify-symbol-dot-pdb-and-source-files-in-the-visual-studio-debugger.md)します。  
  
> [!IMPORTANT]
> 読み取るため *.pdb*ファイルがファイル内の任意のコードの実行、サーバーを信頼するかどうかを確認します。  
  
- **ソース サーバー診断メッセージを出力ウィンドウに**: この設定は、診断の表示をオンにソース サーバー サポートを有効にします。  
  
- **部分信頼アセンブリ (マネージのみ) のソース サーバーを許可する**: ソース サーバー サポートを有効にすると、この設定は、部分信頼アセンブリのソースを取得しないの既定の動作をオーバーライドします。  

- **常に確認なしで信頼されていないソース サーバー コマンドを実行**: ソース サーバー サポートを有効にすると、この設定は、信頼されていないコマンドを実行するときに確認メッセージの既定の動作をオーバーライドします。

**ソース リンクのサポートを有効にする**: のソース ファイルをダウンロードする Visual Studio デバッガーに指示 *.pdb*ソース リンクの情報を含むファイル。 ソース リンクの詳細については、次を参照してください。、[ソース リンク仕様](https://github.com/dotnet/core/blob/master/Documentation/diagnostics/source_link.md)します。

> [!IMPORTANT]
>  ソース リンクは、http または https を使用してファイルをダウンロードは、あるため、信頼できる確認、 *.pdb*ファイル。  
  
- **ソース リンク要求はすべての Git Credential Manager の認証にフォールバック**: ときのソース リンクのサポートが有効になっていると、ソース リンク要求を選択し、認証では、Visual Studio は失敗し、Git Credential Manager を呼び出します。

**ブレークポイントおよび現在のステートメント (C++ のみ) の行全体を強調表示**: デバッガーには、ブレークポイントまたは現在のステートメントが強調表示、ときに行全体が強調表示されます。  
  
**元のバージョンと正確に一致するソース ファイルが必要です。**: ソース ファイルがソース コードをデバッグする実行可能ファイルをビルドするために使用のバージョンと一致することを確認するデバッガーに指示します。 バージョンが一致しないときに対応するソースを検索するように求められます。 一致するソースが見つからないときは、デバッグ時にソース コードが表示されません。 
  
**イミディ エイト ウィンドウにすべてのテキストを出力ウィンドウをリダイレクト**: すべてのデバッガーに通常表示されるメッセージを送信、**出力**ウィンドウ、**イミディ エイト**ウィンドウ代わりにします。  
  
**オブジェクトの生の構造体を変数ウィンドウに表示**: すべてのオブジェクト構造ビューのカスタマイズをオフにします。 ビューのカスタマイズの詳細については、次を参照してください。 [.managed オブジェクトのカスタム ビューを作成](../debugger/create-custom-views-of-dot-managed-objects.md)です。  
  
**モジュールの読み込み (マネージドのみ) に JIT 最適化を抑制する**: モジュールが読み込まれ、デバッガーがアタッチされている間、JIT のコンパイル時に、マネージ コードの JIT 最適化を無効にします。 最適化を無効にした場合、一部の問題のデバッグは簡単になりますが、パフォーマンスは低下します。 [マイ コードのみ] を使用しているときに JIT 最適化を抑制すると、非ユーザー コードがユーザー コード ("マイ コード") として表示される可能性があります。 詳細については、次を参照してください。 [JIT の最適化とデバッグ](../debugger/jit-optimization-and-debugging.md)します。

**ASP.NET (Chrome、Edge、および IE) の JavaScript のデバッグを有効にする**: ASP.NET アプリ用のスクリプト デバッガーを使用します。 Chrome で初回使用時に、インストールされている Chrome の拡張機能を有効にするには、ブラウザーに署名する必要があります。 従来の動作に戻すには、このオプションを無効にします。    

**UWP JavaScript アプリ (試験段階) の Edge 開発者ツールを有効にする**: エッジでの UWP JavaScript アプリの開発者ツールを有効にします。

**ASP.NET のレガシ Chrome JavaScript デバッガーを有効にする**: ASP.NET アプリ用のレガシ Chrome JavaScript スクリプト デバッガーを使用します。 Chrome で初回使用時に、インストールされている Chrome の拡張機能を有効にするには、ブラウザーに署名する必要があります。

**Visual Studio を管理者として実行中の Chrome JavaScript のデバッグを起動する実験的な方法を使用して、**: JavaScript のデバッグ中には、Chrome を起動する新しい方法をお試しに Visual Studio の通知。

**Dll のエクスポート (ネイティブのみ) を読み込む**: dll エクスポート テーブルを読み込みます。 DLL エクスポート テーブルのシンボル情報は、対応するシンボルのない Windows メッセージ、Windows プロシージャ (WindowProc)、COM オブジェクト、マーシャリング、DLL を操作する場合に役立ちます。 DLL エクスポート情報を読み取ると、オーバーヘッドがある程度発生します。 そのため、既定ではこの機能はオフになっています。  
  
DLL のエクスポート テーブル内で使用できるシンボルを確認するには、`dumpbin /exports` を使います。 シンボルは、すべての 32 ビット システムの DLL に使うことができます。 `dumpbin /exports` の出力を参照すると、英数字以外の文字を含む、正確な関数名を確認できます。 この情報は、関数にブレークポイントを設定するときに使用します。 DLL エクスポート テーブルの関数名は、デバッガーの他の場所で表示されるとき、切り捨てられることがあります。 関数は呼び出し順に表示され、現在の関数 (入れ子の一番内側) が先頭に表示されます。 詳細については、「 [dumpbin /exports](/cpp/build/reference/dash-exports)」を参照してください。  
  
**表示する並列スタックの図を上下**: スタックがで表示する方向を制御、**並列スタック**ウィンドウ。
  
**書き込まれたデータは、値を変更していない場合は、GPU メモリ アクセス例外を無視する**: データが変更されなかった場合、デバッグ中に検出された競合状態は無視されます。 詳細については、次を参照してください。 [GPU コードのデバッグ](../debugger/debugging-gpu-code.md)します。  
  
**マネージ互換モードを使用して、**: デバッグ エンジンをレガシ バージョンをこれらのシナリオを有効にすると既定値を置き換えます。  
  
- 独自の式エバリュエーターを提供する C#、VB、または F# 以外の .NET Framework 言語を使用しています (これは C++/CLI を含みます)。  
  
- 混合モード デバッグ時に C++ プロジェクトのエディット コンティニュを有効にします。  
  
> [!NOTE]
> マネージ互換の選択モードには、既定のデバッグ エンジンでのみ実装されているいくつかの機能が無効になります。 

**使用して、従来のC#と VB の式エバリュエーター**: デバッガーが Visual Studio 2013 を使用してC#Visual Studio 2015 の Roslyn ベースの式エバリュエーターではなく、または VB の式エバリュエーター。    
  
**安全でない可能性のあるプロセス (マネージドのみ) に対してカスタム デバッガー ビジュアライザーを使用する場合に警告**: 実行する可能性があるために、デバッグ対象のプロセスでコードを実行しているカスタム デバッガー ビジュアライザーを使用しているときに Visual Studio 警告安全でないコードです。  
  
**Windows デバッグ ヒープ アロケーター (ネイティブのみ) を有効にする**: ヒープ診断を向上させるために windows デバッグ ヒープを使用します。 このオプションを有効にすると、デバッグ パフォーマンスに影響を与えます。  
  
**XAML の UI デバッグ ツールを有効にする**: デバッグを開始するときに、ライブのビジュアル ツリーとライブ プロパティ エクスプ ローラー ウィンドウが表示されます (**F5**)、サポートされているプロジェクトの種類。 詳細については、次を参照してください。[デバッグ中に XAML の検査プロパティ](../debugger/inspect-xaml-properties-while-debugging.md)します。  
  
- **Live Visual Tree で選択した要素をプレビューする**: コンテキストが選択されている、XAML 要素が選択されても、 **Live Visual Tree**ウィンドウ。  
  
- **アプリケーションでランタイム ツールを表示する**: を示しています、 **Live Visual Tree**ツールバーのデバッグ中に XAML アプリケーションのメイン ウィンドウでコマンド。 このオプションは、Visual Studio 2015 Update 2 で導入されました。

## <a name="options-available-in-older-versions-of-visual-studio"></a>Visual Studio の以前のバージョンで使用できるオプション

以前のバージョンの Visual Studio を使用している場合、追加のオプションが存在する可能性があります。

**例外処理アシスタントを有効にする**: マネージ コードは、例外処理アシスタントを使用するとします。 Visual Studio 2017 では、例外ヘルパーには、例外処理アシスタントが置き換えられます。

**ハンドルされない例外でコール スタックをアンワインド**: により、**呼び出し履歴**にロールバックする呼び出しスタック、ポイント、ハンドルされない例外が発生する前にウィンドウ。 

**警告 (ネイティブのみ) の起動にシンボルが見つからない**: をデバッガー シンボル情報いないプログラムをデバッグするときに警告 ダイアログ ボックスが表示されます。 

**起動時にスクリプトのデバッグが無効になっているかどうかに警告**: スクリプト デバッグが無効で、デバッガーを起動すると、警告ダイアログ ボックスが表示されます。

**ネイティブ互換モードを使用して、**: このオプションを選択すると、デバッガーが、新しいネイティブ デバッガーではなく、Visual Studio 2010 のネイティブ デバッガーを使用します。  
  
- 新しいデバッグ エンジンは .NET C++ 式の評価をサポートしていないため、.NET C++ コードをデバッグするときにこのオプションを使用します。 ただし、ネイティブ互換モードを有効にすると、現在のデバッガーの実装に依存している多くの機能が無効になります。 レガシ エンジンに多くのビジュアライザーがないなどの組み込み型など、`std::string`で Visual Studio 2015 のプロジェクト。   このような場合の最適なデバッグ エクスペリエンスの Visual Studio 2013 プロジェクトを使用します。
