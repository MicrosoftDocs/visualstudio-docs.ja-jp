# UI テキストと Visual Studio のヘルプ
[!INCLUDE[vs2017banner](../../code-quality/includes/vs2017banner.md)]

##  <a name="BKMK_UITextAndTerminology"></a> UI テキストと用語  
 包括的テキストは、効果的な UI にとって不可欠です。 ソフトウェア ユーザーを読み取る傾向があるラベルを最初に、つまり、手元の作業を完了することに最も関連します。 静的テキストを少ない頻度で読み取られます。 この適切な順番では、UI の読み取り後に、ウィンドウ全体のクイック スキャンでの作業セッションを開始するユーザーの計画します。  
  
1.  中央の対話型のコントロール  
  
2.  コミット ボタン  
  
3.  対話型のコントロールの場所にあります。  
  
4.  主な手順  
  
5.  補足的な説明  
  
6.  ウィンドウのタイトル  
  
7.  本文の他の静的テキスト  
  
### UI テキストの使用パターン  
  
#### タイトル バーのテキスト  
 タイトル バーのテキストには、UI を生成するコマンドと一致する必要があります。  
  
#### 説明のテキスト \(ヘルパー テキスト\)  
 一部のダイアログで、ウィンドウまたはページの対処方法を説明する著名な主要手順を提供することをお勧めします。 これとも呼ば"ヘルパー text"です。  
  
##### ヘルパーのテキストのスタイル ルールを作成  
  
-   明らかな原因を説明しません。 本当に必要な場合を除きでは、説明のテキストは含まれません。  
  
-   説明のテキストでは、常にダイアログの上部に配置され、実行するタスクを参照してください。  
  
-   ユーザーに必要な作業について説明する正確に。 過剰な通信と冗長性を回避します。  
  
-   各ウィンドウを確認し、重複する単語および文を削除します。  
  
-   短い説明のテキストを保持します。 詳細については特定のユーザーまたはシナリオのために必要な場合は、オンラインの概念の詳細なトピックへのリンクを提供します。  
  
-   キャッシュの重みを保持して、すべての単語が必要なように、テキストを記述します。  
  
-   既存の Microsoft ガイダンスに従う [ユーザー インターフェイス テキスト](https://msdn.microsoft.com/en-us/library/windows/desktop/dn742478\(v=vs.85\).aspx) と [スタイルとトーン](https://msdn.microsoft.com/en-us/library/windows/desktop/dn742477\(v=vs.85\).aspx)します。  
  
#### 補足手順  
 補足的な手順については、ユーザー コントロールを理解するか、またはグループ化の制御に役立つ追加情報を提供します。 ヒント テキスト入力コントロールを予期しているどのような形式を理解するために必要な可能性があります。 補足の手順を慎重に使用します。 ここでは、ユーザーが作成するかを選択した場合の動作を完全に理解しない可能性が高い場合に、これらを予約します。

#### ヒント  
 多くの場合、説明のテキスト UI でのインプレースでの位置には長過ぎる場合がありますか、またはに慣れているユーザーの煩雑さのように感じて、新しいユーザーにのみ便利です。 この場合、ヒントの下のツールヒントとして、または情報の説明テキストを配置してください。  
  
 ヒントは、顕著に関連して、特定のヒントのアイコンはまだ目障りを使用する必要がありますが、コントロールの近くに配置する必要があります。  
  
 ![InfoTip in Visual Studio](../../extensibility/ux-guidelines/media/0601-d_infotip.png "0601\-d\_InfoTip")

##### ヒントのスタイル ルールの書き込み  
  
-   完全な文としてヒントを作成します。 特定の動詞、文の場合と句点を必要とします。  
  
-   ヒントを使用して、メイン インストラクションまたは情報を補完します。 別の単語を使用して、主要なアイデアをもう一度説明だけが場合、されるヒントで確認する必要はありません。  
  
-   ヒントを簡潔にしてください。 小規模の単語とサフィックスなしを使用して日常的な言語をサポートし、ユーザーのことをお勧めします。  
  
-   既存の Microsoft ガイダンスに従う [ユーザー インターフェイス テキスト](https://msdn.microsoft.com/en-us/library/windows/desktop/dn742478\(v=vs.85\).aspx) と [スタイルとトーン](https://msdn.microsoft.com/en-us/library/windows/desktop/dn742477\(v=vs.85\).aspx)します。  
  
#### コントロールのラベル  
 コントロールのラベルと必要があります、短く簡潔で、次の [コントロールについては、Windows デスクトップ](https://msdn.microsoft.com/en-us/library/windows/desktop/dn742399\(v=vs.85\).aspx)します。  
  
 コントロール ラベルの書式と、UI 内での配置の詳細についてを参照してください [Visual Studio のレイアウト](../../extensibility/ux-guidelines/layout-for-visual-studio.md)します。  
  
#### ヘルプ リンク  
 説明のテキスト内で、または、UI の本文に、ヘルプのリンクを配置するかできます。 ヘルプへのリンクまたは内部のダイアログ ボックスを起動できます。  
  
##### ヘルプのリンクの visual スタイルのルール  
  
-   ハイパーリンクの適切な環境の色を使用します。 正しくスタイル設定されたハイパーリンク クリックされたときに、赤はについて簡単に点滅しません。 これを表示する場合は、環境の色が使用されていないことを示すです。  
  
-   下線は、ホバーまたは段落内のリンクが埋め込まれたときにのみ使用する必要があります。  
  
-   ハイパーリンクのスタイルを表示と対話の詳細については、ボタンやハイパーリンクを参照してください。  
  
##### ヘルプのリンクのスタイル ルールを作成  
  
-   ダイアログ ボックスを起動するときに、省略記号の水準を維持します。 タスクには、その他の UI が必要な場合は、省略記号ボタンを移動するための省略記号なし。  
  
     ![Help link in Visual Studio](~/docs/extensibility/ux-guidelines/media/0601-e_helplink.png "0601\-e\_HelpLink")  
  
     **省略記号 \(...\) ヘルプでは、リンクは、その他の UI が必要なタスクを示します。**  
  
-   ユーザーの目的ではないために、リンクは「について、」で開始しないようにします。 ユーザーは、特定の質問に答える、全般的な教育を受け取りませんができます。  
  
-   語句のヘルプにリンク、質問を投稿すると、トピックに応答することができるようにします。  
  
     正しくありません。  
     「Windows Azure モバイル サービスの価格の詳細情報について説明」  
  
     修正できます。  
     「どのような価格オプションは、Windows Azure モバイル サービスの利用可能な」でしょうか。  
  
-   使用しないで *をクリックして* リンク テキストにします。  
  
-   リンクの単語のみ"here"ことはありません。 これは、ハイパーリンク付きの単語のみを音声一部のスクリーン リーダーにとって問題であります。  
  
     正しくありません。

     修正できます。  
     「どのような価格オプションは、Windows Azure モバイル サービスの利用可能な」でしょうか。  
  
-   ヘルプ リンクの正しい書き込みスタイルの詳細については、次を参照してください。、 [ヘルプについては、Windows デスクトップ](https://msdn.microsoft.com/en-us/library/windows/desktop/dn742494\(v=vs.85\).aspx)します。

## エラー メッセージ  
  
### 概要  
 エラーが発生します。 により、可能なエラー メッセージが適切な最初の手順は、ユーザーの機能に制限を設定します。 ただし、エラーが発生した場合、適切に記述されたエラー メッセージは、問題を回避するための長期的に移動できます。 エラー メッセージは、ほぼ間違いなく最も重要な種類の同期を解決する必要がある問題があるため、ユーザーに表示される通知のいずれかです。 不適切なエラー メッセージをエラーの原因を判断するために、考えられる解決策をユーザーがままにします。  
  
 ユーザーは、過剰に注意を払い、またはユーザーに値を追加する書き込みのみに必要なメッセージが発生するので、エラー メッセージの混乱を招く停止可能性があります。 メッセージが通知するだけの場合は、別の表示形式を使用します。  
  
### エラー メッセージを作成するためのルール  
  
-   エラー メッセージを構築するときに、対象ユーザーの適切なエラー レベルを選択します。 該当する場合は、ユーザーが行える操作を提供する簡単な概要のことを目指します。 ユーザーが知る必要がないものはすべての状態をしないでください。  
  
-   建設的な支援を提供します。 処理命令を含むエラー メッセージを読み取って簡単です。  
  
-   二重否定を使用しないでください。  
  
-   両方自動実行し、手動の文法およびスペル チェックを記述するすべてのエラー メッセージを確認します。  
  
-   複雑なエラー メッセージの順次通信しないようにします。 エラー メッセージを F1 フックアップを使用しないでください。 メッセージ自体はすれば十分です。  
  
-   適切なアイコンを使用します。  
  
-   質問を簡単に理解し、"Delete"と「キャンセル」など、クリア、選択のボタンの使用  
  
-   警告には、先に進むの結果がどのようになるかについて、明確にします。 ボタンには、その結果を示す必要があります。  
  
-   エラーの場合、問題を解決するユーザーが実行できるについて説明します。 ボタンの操作をするか「閉じる」と答えて エラー メッセージには、\[OK\] ボタンを使用しないでください。  
  
-   いくつかの質問をエラー メッセージを構築するときに考えてみてください。  
  
    -   ユーザーがだけで、このエラーと共に、問題を解決する方法を見つけることができますか。  
  
    -   ユーザーは、このエラーと同じボキャブラリを使用でしょうか。  
  
    -   このエラーがあいまいです。 ですが複数ある状況での共有か 場合は、どの操作を行う手順をユーザーに必要なソリューションにでしょうか。  
  
#### ビルド エラー  
 Visual Studio が、ソフトウェア開発ツールであるため、多くのコンポーネントのコンパイル、変換、または開発者の手間をバイナリ形式に変換する手順のエンコードがあります。 これらの変換には、コンパイラが不適切に作成されたファイルを処理できないとき、またはコンパイラ オプションが正しく設定されなかったときにエラーが発生します。  
  
 Visual Studio のユーザーは、開発時間がビルド エラーの解決の数は非常に費やすことができます。 エラーがある依存関係やとエラー メッセージが適切に作成するが難しく、エラーの原因を明らかにするために、この解決時間が増加します。  
  
 最適なビルド エラー オートコンプリート機能を提供しない Visual Studio では、まず初めに、出現するものは、IntelliSense を示す波線します。 スキーマの検証コントロールと同様のツールは、同じ種類のフィードバックを提供します。 これらのメカニズムは、事前にビルド エラーの可能性を緩和すること、適切な形式のコードを作成するユーザーを説明します。  
  
 Visual Studio には、ツール ウィンドウ、ユーザーが読み取るし、ドキュメント ウィンドウで発生したエラーのナビゲートが用意されています。 ユーザーがすばやく大量のコードを移動し、問題の場所に直接移動できるように、キーボード ショートカットが提供されます。 Visual Studio では、各ビルド エラーできるように、ユーザーが、エラーの詳細については、ヘルプ トピックに直接移動して、特定のヘルプ キーワード\/コンテキスト ID に関連付けることもできます。  
  
 明確で正確なビルド エラーを記述します。  
  
-   **プレーンな言語を使用して** ほとんどまたはまったくないコンパイラの専門用語の問題を説明します。 ビルド エラーのテキストを過度に技術的なすることはできません。  
  
-   **考えられる原因を説明します。**たとえば、"プロパティと値の間にコロンがありません、' \(プロパティ\): \(値\)' の宣言です"。  
  
-   修正候補の詳細についてを説明します。 十分な余裕がない場合は、対応するヘルプ トピックに追加の詳細情報を格納することがあります。  
  
### 適切に記述されたエラー メッセージの構成  
  
#### エラー メッセージをシェル ダイアログ サービスを使用します。  
 Shell ダイアログ サービスを使用すると、メッセージの外観を制御できます: 特定のフォント — 個々 の要素に大幅な変更なし。 使用して、 **IErrorInfo** メカニズムに報告を使用して **IVsUIShell::SetErrorInfo\/ReportErrorInfo**します。  
  
#### 通知が有効かつ適切なプレゼンテーションを選択します。  
 早急な対応が \(同期通知\) のデータの損失を防ぐために必要な場合は、重大な警告があるモーダル ダイアログ ボックスを使用します。 不適切な結果を招く可能性を読まずにメッセージを終了中の場合は、重要なアイコンが予約されています。 データの損失は、アラーム レベル応答を必要とする重大な状況です。 "重大"アイコンの使いすぎでは、ユーザーがその重要性を desensitizes します。 エラー メッセージは、本質的に情報がある場合は、モーダル ダイアログ ボックス \(非同期通知\) に代わる方法を検討します。  
  
#### 技術的な説明ではなく、問題が発生した理由のクリーン、簡潔な説明を提供します。  
 説明に技術的な詳細を持つユーザーの過負荷をかけるようにしてエラー メッセージを無視する可能性が高くなります。 適切なメッセージの例:  
  
-   「要求されたファイルを開けません。」  
  
-   「インターネットに接続できません。」  
  
#### 問題を解決する方法についてを説明します。  
 ユーザーの提案、問題を解決する方法を提供します。 正直なユーザーに提案がない場合です。 テクニカル サポート、コミュニティのサポートなどの他のオンライン ソースへの直接のリンクを提供します。 問題に関連する特定のオンライン情報をユーザーに示すしようとしてください。 エラー ID は、ユーザーをその特定のエラーについてのディスカッション スレッド リンクを検討してください。 適切なメッセージの例:  
  
-   「インターネットに接続しているし、この操作を再試行していることを確認してください。」  
  
-   「ファイルが存在して、それを開くアクセス許可があることを確認してください。」  
  
#### 短期的およびポイントには、メッセージを記述します。  
 エラー メッセージが表示できる、によりを説明して、ソリューションが提供も好きである場合も無視されます。 1 つのソリューションでは、詳細ボタンをプログレッシブの公開を使用します。 たとえば、短い説明\/ソリューションを提供し、詳細については、詳細ボタンをクリックします。 ユーザー エラーの詳しい情報を読み取る場合、そうできます。  
  
 メッセージの言語になります。  
  
-   **ドメインの適切なします。**ユーザーが理解言語を使用します。 お客様は、開発者は、多くの場合はありません、コンテキストと用語があります。  
  
-   **特定します。**あいまいな表現を回避し、関連するオブジェクトの特定の名前と場所を指定します。 たとえば、エラー メッセージなど「文字が有効な」がない有効です。 検索を開始するでしょうか。 「ファイルが見つかりません。」 どのファイルですか。  
  
-   **お勧めします。**ばかげていると感じるようにしたり、ユーザーを非難しないでください。 悪意のあるまたは有害な言語を回避 \(強制終了、実行、終了、致命的な不正な\)。 大文字のテキストが誇張表現として解釈されるよう多くの場合と、読み取ることができないしないようにします。 ユーモアを使用しないでください。  
  
-   **修正します。**\(アルファ\) であっても、正しいスペルおよび文法を使用します。 入力ミスは、プロおよび厄介なです。  
  
-   **適切なコンテキストに応じて切り替えたりします。**適切なボタンのテキストを使用します。 \[OK\] ボタンを回避し、代わりに"Continue"または「はい\/いいえ。」  
  
### エラー メッセージの例

## ヘルプへのアクセス  
  
### 概要  
 MSDN のドキュメントの他は、Visual Studio ユーザーは、UI の中にユーザーを支援するためにいくつかのアクセス ポイントを持っています。 これらのアクセス ポイントが常に使用可能であることを確認するには、機能チームは、環境によって提供されているヘルプ システムを利用する必要があります。 これらのアクセス ポイントは次のとおりです。  
  
-   **ダイアログ ボックスの教育および補足のテキストです。**方向か、詳細については、表面またはヒント アイコンの上のホバー時の使用可能な UI のいずれかが静的なテキスト。  
  
-   **F1 ヘルプ** \(エディターのみ\)。 Visual Studio エディター内で、ユーザーは、いつでも F1 キーを押してが表示されます、現在の選択に固有のヘルプ トピックを信頼できます。 F1 キーに関連付けられているトピックが適切なや役に立つことを確認します。  
  
-   **ヘルプ トピックへのハイパーリンクです。**ダイアログ、ツール ウィンドウで、またはの詳細については、テクノロジ、機能、またはタスクを実行する方法については、ユーザーを支援するトピックを起動するデザイン サーフェイス内のハイパーリンクです。  
  
-   **スマート タグと構築ダイアログなどのヘルパー UI メカニズムです。**これらのメカニズムは、UI 要素を理解するうえで、ユーザーを支援またはスマート タグまたはビルダー ダイアログなどのタスクを容易にします。  
  
-   **UI のヘルプ\] ボタン** \(非推奨\)。 関連の F1 ヘルプ トピックにアクセスできるように、タイトル バーに表示されるインジケーター。

##### 環境を使って作成されたダイアログ ボックス  
 多くのシェル ダイアログが作成された、 **VBDialogBoxParam** 関数です。 移行時に支援するためにこの共有関数が更新された、 **ヘルプ** \] ボタンをクリックして、ダイアログ ボックスから、 **ですか?** は旧バージョンとアーキテクチャを維持したままボタン互換性があり、拡張可能なです。  
  
 具体的には、 **VBDialogBoxParam** 関数は参照 id を持つボタン用のダイアログ テンプレート **IDHELP** \(9\) か、ラベル **ヘルプ** または **& ヘルプ**します。 非表示で、\[ヘルプ\] ボタンが見つかった場合、 **WS\_EX\_CONTEXTHELP** 配置ダイアログ ボックスにスタイルを追加、 **ですか?** \] ダイアログ ボックスのタイトル バー ボタンです。  
  
 ダイアログ proc がスタックにプッシュしてという名前の前処理ダイアログ proc でダイアログを起動、ダイアログ ボックスが作成されると、 **DialogPreProc**します。 ときに、 **ですか?** \] ボタンをクリックすると、送信、 **位置である** の **SC\_CONTEXTHELP** \] ダイアログ ボックス。**DialogPreProc** このコマンドをキャプチャし、それを変更して、 **WM\_HELP** は元のダイアログ プロシージャに渡されるメッセージ  
  
 多くの環境が作成したダイアログ ボックスでは、ダイアログ ボックスのヘルプ ボタンがあります。 \[ヘルプ\] ボタンは、自動非表示になりのみダイアログが表示されたら、 **?** works のボタンをクリックします。 場合、 **?** このソリューションが、元のヘルプ ボタンにすばやく移動することができます\] ボタンがこれまで削除されるか、Windows で変更、します。  
  
 このソリューションは、バグの原因となる 4 つの前提条件です。  
  
-   ダイアログ ボックスの \[ヘルプ\] ボタンを **IDHELP** \(9\)。  
  
-   \[ヘルプ\] ボタンが表示されていない場合、ダイアログ ボックスが正しい。
