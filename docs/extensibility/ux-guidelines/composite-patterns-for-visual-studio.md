##  <a name="a-namebkmkdatavisualizationa-data-visualization"></a><a name="BKMK_DataVisualization"></a> データの視覚化  
  
### <a name="overview"></a>概要  
 グラフは、集計および意思決定を強化するためにデータを視覚化する視覚的です。 多数のデータが、ほとんど意味がどのような注目に値するとアクションを参照してくださいで直面しているユーザーに役立ちます。  
  
 ユーザーは、次の条件のいずれかに当てはまる場合に、グラフから活用できます。  
  
-   グラフに役立つユーザーが対処するタスクを識別するでしょうか。  
  
-   グラフでは、潜在的な変更の結果を予測するユーザーを有効にします。  
  
-   グラフをユーザーの傾向を発見し、パターンを識別するやすくなりますか。  
  
-   グラフは、ユーザーがより良い意思決定を許可しますか。  
  
-   グラフに役立つ可能性のあるユーザー指定のコンテキストで特定の質問に答えるでしょうか。  
  
#### <a name="general-rules-for-charts"></a>グラフの一般的な規則  
  
-   [ラベル データ明確にします。 説明せず図とは非常に画像です。  
  
-   傾斜の縦横比を回避する 0 から軸を起動します。 行の長さとバーのサイズは、データ ポイント間の関係を理解するための重要な視覚的手掛かりです。  
  
-   グラフ、いないインフォ グラフィックを作成します。 インフォ グラフィックをデータの芸術的な表現し、その主な目的は、visual ストーリーテ リングします。 グラフは、視覚的に魅力的ですが (および必要があります) できますが、データがそれ自体の話です。  
  
-   Skeumorphism、棒グラフの画像、コントラスト hashmarks およびその他のインフォ グラフィックな調整を回避します。  
  
-   装飾的な要素としては、3 D 効果を使用しません。 場合にのみを使用してこれらの情報を理解するユーザーの機能に本当に不可欠です。  
  
-   複数の 2 つの色と、この種類のグラフ読みにくくなり、正しく解釈と、複数の直線と塗りつぶしの使用を避けます。  
  
-   概念を理解することや、データと対話する唯一の手段としてグラフ (または任意の図) を使用することはできません。 これは、視覚障碍があるとユーザーの問題を表示します。  
  
-   ページ上の不要なまたは装飾的な要素としては、グラフを使用しないでください。 つまり、値またはヘルプのすべてのユーザーの問題を解決するグラフが追加されない場合、使用しないことです。  
  
### <a name="chart-types"></a>グラフの種類  
 Visual Studio で使用されるグラフの種類には、横棒グラフ、折れ線グラフ、リング グラフまたは「ドーナツ グラフ」、タイムラインと呼ばれる、変更された円グラフ プロット (「クラスター グラフ」)、およびガント チャートの散布図です。 各グラフの種類は、さまざまな種類の情報を通信するために役立ちます。  
  
### <a name="other-charting-considerations"></a>その他のグラフに関する考慮事項

##  <a name="a-namebkmkonobjectuia-on-object-ui-and-peeking"></a><a name="BKMK_OnObjectUI"></a> オブジェクトの UI およびピークする方法  
 このセクションでは、ピーク、コードのピーク ビューとも呼ばれるオブジェクトの UI の Visual Studio に固有の型にコンテキストを説明します。  
  
### <a name="overview"></a>概要  
  
-   オブジェクトの UI ようにユーザー詳細または対話機能せず、メインのタスクから低下します。  
  
-   Visual Studio でのオブジェクトの UI のメインのパターンは「注意時点での情報」と呼ばれます  
  
-   Visual Studio でのオブジェクトの UI は、インラインまたは浮動小数点とか持続性の一時的なものです。  
  
    -   コードのピーク] ビューで、Visual Studio ではオブジェクトの ui 型は、インラインと永続的なです。  
  
    -   CodeLens では、Visual Studio で、オブジェクトの UI の型が浮動小数点と一時的です  
  
 コンテキストの切り替えを移動するその他のコンテンツまたは別の開発者が多くの場合、必要、コードの断片のしくみを理解すること、またはそのコードの詳細を検索] ウィンドウ。 ようなコンテキスト移行は、メイン ウィンドウのままにする場合は、ユーザーに、元のタスクにフォーカスが失われることができますので、混乱を招くことができます。 さらに、元のコンテキストの背面が困難ですが、他の UI に隠されるため元のコード ウィンドウを切り替えます。 原因となった場合に特にできることを取得します。  
  
 オブジェクトの UI に依存して「時に、注意情報」という名前のパターン これらのメッセージ、ポップアップ ウィンドウとダイアログ ボックスは、その主なタスクにフォーカスを失うことがなく、明確化または対話機能を追加する、関連する追加の情報をユーザーに付与します。 オブジェクトの UI の例には、ユーザーが通知領域のアイコン、スペル ミスの単語と Visual Studio 2013 で導入されたピーク ビューの下に赤い波線上にポインターを置いたときに表示されるポップアップ ウィンドウが含まれます。  
  
### <a name="decision-points"></a>意思決定ポイント  
 Visual Studio 内で、注意の時点での情報には、このパターンを使用するいくつかの方法があります。 適切なメカニズムを選択して、一貫性のある予測可能な方法で実装するエクスペリエンス全体に不可欠です。 それ以外の場合、コンテンツ自体からフォーカスが損なわれることを混乱を招くまたは一貫性のない経験を持つユーザーに提示する可能性があります。  
  
#### <a name="relationships-between-master-and-detail-content"></a>マスター/詳細コンテンツ間のリレーションシップ  
 注意の時点で情報を使用してリレーションシップを表示するコンテンツの間でユーザーが (「マスター」コンテンツ) に焦点を合わせ、追加である関連するコンテンツ (「詳細」のコンテンツ)。 このパターンでは詳細コンテンツは、マスターのコンテンツの近くに表示可能なユーザーは連携し、コンテンツを明確に関連します。 補足情報やマスター コンテンツを膨大にまとめることができない情報は、ツール ウィンドウなどの別のパターンに従う必要があります。  
  
-   **常に** マスター コンテンツの近くに詳細コンテンツを表示します。  
  
-   **常に** 詳細コンテンツにユーザーがマスター コンテンツに焦点を当ててができるようにもことを確認します。 多くの場合、これを実現するために最適な方法は、可能な限りマスター コンテンツに近いとして詳細コンテンツをレンダリングします。 これは、マスターのコンテンツの横にあるポップアップ ウィンドウに詳細コンテンツをレンダリングすることによって、またはマスター コンテンツの下にある詳細コンテンツ インラインをレンダリングすることによって実行できます。  
  
-   **決して** 時に、十分な注意はマスター コンテンツからユーザーが情報を使用します。 ユーザーは、詳細コンテンツを個別に表示する必要がある場合は、ユーザーがこれを行うできる明示的な操作を公開します。  
  
#### <a name="design-details"></a>設計の詳細について  
 オブジェクトの UI が適切な選択を確認した後は、次の 4 つのメインのデザインに関する考慮事項があります。  
  
1.  **永続性:** コンテンツできると予想される揮発性か一時的なものでしょうか。   
    ユーザーは情報を参照するかとの対話を表示したままにするでしょうか。 または、ユーザー情報にすばやく概要し、その主なタスクを続行するか。  
  
2.  **コンテンツの種類:** コンテンツなります情報、実践的なまたはナビゲーションでしょうか。   
    ユーザーにマスター コンテンツに関する追加情報が必要ですか。 ユーザーは、マスターのコンテンツに影響を与えるタスクを完了する必要があるでしょうか。 または、ユーザーは他のリソースに送信する必要がありますか。  
  
3.  **インジケーターの種類:** アンビエント インジケーターことに意味がでしょうか。   
    情報は、実用的な方法で集計してマスター コンテンツを過負荷なしで表示でしょうか。  
  
4.  **ジェスチャ:** を呼び出すし、UI を閉じるに使用されるジェスチャーが、何でしょうか。   
    方法はユーザー詳細内容を表示し、送信、離れたでしょうか。 一時的および永続性のある状態を切り替える固定などのジェスチャを追加する値はありますか。  
  
 各これら 4 つの意思決定ポイントは、オブジェクトの UI の主要コンポーネントで影響を与えます。

#### <a name="container-content-presenter-type"></a>コンテナー (コンテンツ プレゼンター) の種類  
 注意の時点でのコンテンツを表示する使用可能な 2 つの主要なオプションがあります。  
  
1.  **インライン:** Visual Studio 2013 コード エディターで導入されたピーク ビューなどのインライン発表者により、これらの新しいコンテンツ領域の既存のコンテンツを移行します。  
  
    -   **必要に応じて** をかなりの時間を参照するか、コンテンツとの対話をかける場合、ユーザーは予想される場合、インライン プレゼンターが存在します。  
  
    -   **回避** インライン発表者のユーザーの予定がある場合は、概要情報を表示し、最小限の中断で主要なタスクを続行します。  
  
2.  **浮動小数点:** 浮動プレゼンターが近くに配置可能な選択したコンテンツが配置されているは、既存のコンテンツのレイアウトを変更しません。 長フローティング コンテンツ パネルを表示するなどのさまざまな方法を利用できる、選択した記号を空白文字が使用可能な最も近いします。  
  
    -   **必要に応じて** ユーザーの予定がある場合は、発表者を浮動する概要情報を表示し、最小限の中断で主要なタスクを続行します。  
  
    -   **回避** をかなりの時間を参照するか、コンテンツとの対話をかけたくはユーザーの予定がある場合は、発表者を浮動表示します。  
  
#### <a name="content-type"></a>コンテンツの種類  
 すべてのオブジェクトの UI コンテナー内に表示されることができますコンテンツには、主に 3 種類があります。 これらの種類の情報の任意の組み合わせを表示できます。 次の 3 つの種類があります。  
  
1.  **情報:** ほとんどオブジェクトの UI コンテナーはある種の情報コンテンツを表示します。 コンテンツは、環境の現在の状態に関する情報を表すことができます、または環境内の将来の潜在的な状態に関する情報を表すことがあります。 たとえば、既存のコードに、リファクタリングなどの特定のコマンドの効果を示す使用可能性があります。  
  
    -   **常に** を表示する情報の正規表現を使用します。 たとえば、コードは、構文の強調表示、完全なコードのようになります、任意のフォントおよびその他の環境設定をユーザーが設定を尊重する必要があります。  
  
    -   **常に** マスター コンテンツとしてその同じ情報が表示される場合に可能になる情報のコンテンツをすべてのアクションをサポートを検討します。 たとえば場合オブジェクトの UI コンテナー内の既存のコードを表示するには、検討してください参照し、そのコードを変更する機能をサポートします。  
  
    -   **常に** 将来の潜在的な状態情報を提示することを表している場合、異なる背景色の使用を検討します。  
  
2.  いくつかのオブジェクトの UI コンテナーは、リファクタリング操作の実行などのマスター コンテンツに対して何らかのアクションを実行する機能を実践的な: です。  
  
    -   **常に** 情報のコンテンツと切り離して実践的なコマンドを配置します。  
  
    -   **常に** を有効にして、該当する場合のアクションを無効にします。  
  
    -   **常に** ダイアログ ボックス内にあるコマンドを表すための標準的なガイドラインを参照してください。  
  
    -   **常に** 、オブジェクトの UI コンテナーの絶対パスで公開されているアクションの数を最小限にしてください。 オブジェクトの UI と対話するには、軽量で高速エクスペリエンス必要があります。 ユーザーの管理オブジェクトの UI コンテナー自体にエネルギーを費やす必要はありません。  
  
    -   **常に** オブジェクトの UI コンテナーが終了または破棄する方法とタイミングを検討してください。 ベスト プラクティスとしてマスターと詳細コンテンツ間のダイアログを終了するアクションも終了オブジェクトの UI コンテナーそのアクションが呼び出されたときにされます。  
  
3.  **ナビゲーション:** 一部オブジェクトの UI コンテナーには、クリックすると、別のウィンドウまたはユーザーの web ブラウザーで、MSDN の記事を開くなどのアプリケーションのリンクが含まれています。  
  
    -   **常に** 「が開いている」ナビゲーション リンクの先頭にユーザーが、予期しないその他のコンテンツを移動できるようにします。  
  
    -   **常に** 実践的なリンクのナビゲーション リンクを区別します。  
  
#### <a name="ambient-indicators-optional"></a>アンビエント インジケーター (省略可能)  
 アンビエント インジケーターには、コードの残りの部分から対照的な色で表示されるテキストを含む、微妙なまたは明らかであるため、波線の下線とスマート タグ アイコンなどの tickler シンボルを含めるを使用できます。 アンビエント インジケーターは、追加の関連する情報の可用性を通信します。 理想的には、有用な情報をユーザーがこれらと対話する必要がなくて提供します。  
  
-   **常に** 注意をそらすしたり、ユーザーを圧倒しないように、アンビエント インジケーターを配置します。 このような方法でアンビエントのインジケーターを配置しない場合は、別のソリューションを検討してください。  
  
-   **常に** に関連付けられているコンテンツにできるだけ近いアンビエント インジケーターを配置します。  
  
-   **常に** それが使用可能情報を要約した指標を作成しようとしています。 他のデータを集計する方法と考えるまたは使用できるデータ項目の数のカウント (たとえば、"3 個の参照"単に「参照」ではなく) を提供することを検討します。  
  
    -   場所、インジケーターのデータおよびは行えません常に計算が表示されますの場合、すぐに、値が計算されるプログレッシブのフィードバックを提供することを検討します。 たとえば、未読の電子メールの数として Windows Phone で電子メールのライブ タイルを更新するときと同様に利用可能なデータの更新を反映して変更をアニメーション化を検討してください。  
  
-   **決して** ユーザーが特定のコンテンツの実行かなりよりも多くのインジケーターを追加します。 アンビエント インジケーターは、ユーザーの介入を必要とせずに便利にする必要があります。 インジケーターでは、オーバーフローおよびその他の管理コントロールに表示する必要がある場合、そのアンビエントが失われます。  
  
#### <a name="gestures"></a>ジェスチャ  
 マスター コンテンツにフォーカスを維持するためにユーザーを許可する際の重要な側面を開き、追加の詳細情報の内容を破棄する右のジェスチャをサポートすることでは。

##  <a name="a-namebkmkselectionmodelsa-selection-models"></a><a name="BKMK_SelectionModels"></a> モデルの選択  
  
### <a name="overview"></a>概要  
 選択モデルは、指定し、ユーザー インターフェイス内で目的の 1 つまたは複数のオブジェクトに対して操作を確認するために使用するメカニズムです。 Visual Studio エディターはドキュメント内の選択の対話のパターンについて説明します。 テキスト エディター、デザイン画面、およびモデリング サーフェスです。  
  
 ユーザーの作業では、Visual studio を示すことが必要でき、Visual Studio からのフィードバックに運用上のユーザーにする必要があります予測どおり応答します。 相違点またはユーザーとユーザー インターフェイスの間のコミュニケーション不足発生する、ユーザーが持つことができるアクションに気付かない予想外の結果。 多くの場合、エラーに気付かれないものがないか、変更されたユーザーに表示されるまでです。 選択範囲のモデルはそのため、ユーザー インターフェイスのデザインの最も重要な部分のいずれかです。 Visual Studio での選択のモデルは Windows で一貫性のあるが、わずかずつ異なるバリエーションがあります。  
  
 Windows では、Visual Studio では、選択範囲のモデルは、相互作用が行われるコンテキストによって異なります。 選択内容は、4 種類のオブジェクトで発生することができます。  
  
-   テキスト  
  
-   グラフィック オブジェクト  
  
-   リストとツリー  
  
-   グリッド  
  
 これらのオブジェクト内では、選択項目の 3 つの種類があります。  
  
-   連続します。  
  
-   非結合します。  
  
-   Region

### <a name="view-selection-models"></a>選択範囲のモデルの表示  
  
#### <a name="tree-view"></a>ツリー ビュー  
 ツリー ビューでの選択には、単純なハイライトが表示されます。 ユーザーは、ノード名やノードのアイコンをクリックすると、ノードが選択されます。 ノードの左側に三角形のグリフを拡張またはコントラクトのツリー コントロールが、例外が 1 つのユーザーの選択には影響しません。 そのノードの子を選択しているときは、親ノードを折りたたみ、時に、選択を親に移動します。  
  
 ![Visual Studio での一般的なツリー ビュー](../../extensibility/ux-guidelines/media/0713-25_treeview.png "0713-25_TreeView")  
  
 **Visual Studio での一般的なツリー ビュー**  
  
 ツリー ビューでは、ツリー内の複数のレベルが異なっても連続したと不整合の選択をサポートできます。 連続したまたは分離された複数の選択肢が表示されているツリー ノードで行う必要があります。 ノードが折りたたまれている場合は、非結合選択は失われ、が折りたたまれているノードが選択範囲を取得します。 この方法では、操作の影響を受けるノードを確認できます。 ノードが折りたたまれている場合になりますがわかりにくかったのどのノードを受ける可能性があります。  
  
 親ノードを選択した場合は、親とそのすべての子に適用する操作の方が効率的な場合がありますが、操作が、親を適用してください。 この場合、チェック ボックスをオンまたは"すべての子に適用する] オプションをユーザーに明示的にすることの確認のダイアログ ボックスなどの操作中にその他の UI を提供します。  
  
##### <a name="renaming"></a>名前を変更します。  
 ツリー内のノードでは、名前の変更をサポートする場合、位置で名前の変更を実行する必要があります。 一括操作では、Visual Studio でのすべてのツリー コントロールで、標準をする必要があります。 すぐにテキストを選択した場合、ユーザー入力を受け付ける準備ができてのノードの名前全体をカバーする、インプレース編集モードを有効にする名前の変更コマンドを提供します。 ノードは、ファイルを表している場合、ファイル名は、拡張機能を含める必要があります。 選択範囲の表示は、ファイル名と拡張子ではなくの本文のみを含める必要があります。

#### <a name="selection-within-lists-and-grid-controls"></a>リストおよびグリッド コントロール内の選択項目  
 リストの選択における主要な概念は、行ベースである、選択範囲には、行全体が行われることを意味が単位として選択されているです。 これに対し、グリッドには、行の他の側面に影響を与えずに選択される特定のセルを許可できます。 グリッドには、入れ子になった行 (のツリー グリッドなど) を選択し、親の行との対話によって選択解除すると、階層の分岐全体を許可するの階層構造も含まれます。 リスト内の選択は、行全体のデータで単純なハイライトの色で表示されます。 現在の編集可能な行またはセル (行のすべてのセルは読み取り専用の場合) を囲む 1 ピクセルの点線でフォーカスが表示されます。  
  
> [!NOTE]
>  **フォーカス** と **選択** はさまざまな概念です。 *フォーカス* UI の要素が対象となることを示しますを明示的に別のオブジェクトを対象の入力を受信するときに *選択* の一連のオブジェクトを後続の操作が場所を受け取ることがでオブジェクトの包含状態を指します。  
  
 連続した、分離された、リストの選択項目がありますまたは地域。 複数の選択肢は、許可されている、連続したと領域 (ボックス) の選択範囲をサポートしながら、非結合選択をサポート常にべきはオプションです。 地域の選択は、リストの本文の空白部分にドラッグすることによって開始されます。  
  
|オブジェクト|選択内容|  
|------------|---------------|  
|リスト|連続します。|(複数選択を許可する) 場合は常にサポートします。|  
|リスト|非結合します。|(複数選択を許可する) 場合は常にサポートします。|  
|リスト|Region|オプションでサポートされます。 リストの本文の空白文字で、マウスをドラッグすることによって開始されます。|  
  
 リストに 1 回クリックすると、クリックが発生した行が選択されます。 ユーザーを一括編集をサポートする] ボックスの一覧のセルをクリックする、埋め込み先編集のセルがすぐにアクティブになります。 それ以外の場合、行全体では、すぐに選択され、強調表示を示しています。  
  
 リストの本文にドラッグすることは次の 3 つのいずれかを行います。  
  
-   一覧でサポートされているし、マウスの一覧が空白では、地域の選択を開始します。  
  
-   一覧のセルまたは行は、ドラッグ ソースがサポートする場合は、ドラッグ アンド ドロップ操作を開始します。  
  
-   現在の行を選択します。

##### <a name="keyboard-support"></a>キーボードのサポート  
 リストとグリッドで選択のキーボードのサポートは、Windows の標準の表記規則に従います。  
  
-   方向キーでは、フォーカスを移動すると、行またはセルを選択すると、リストを移動します。  
  
-   Shift キーを押しながら矢印は、方向キーの方向に連続して選択を実行します。  
  
-   Ctrl + space キーを切り替えます続いてを追加して、非結合選択を作成する、選択範囲からリスト アイテムの削除と矢印です。  
  
-   入れ子構造の階層を含むグリッドの右方向キーが親の行を展開し、左方向キーでは、1 つが折りたたまれます。  
  
-   Tab キーでは、セルが編集可能な場合に、現在の行セル間でフォーカスを移動します。  
  
-   Enter キーは、リスト内の項目の既定のコマンドを実行する (多くの場合、 **開く**)。  
  
-   F2 キーは、現在選択されているセルの編集をインプレースでアクティブにします。
