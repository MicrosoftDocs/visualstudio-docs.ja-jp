## <a name="terms"></a>用語  
 アドイン  
 ユーティリティアプリケーション、ドライバー、またはプライマリアプリケーションに追加されたその他のソフトウェア。 Visual Studio 統合開発環境 (IDE: integrated development environment) では、アドインは IDE の機能を拡張するオートメーションベースのアプリケーションです。  
  
 オートメーション モデル  
 以前のバージョンの Visual Studio では、機能拡張モデルとして知られていたオートメーションモデルは、IDE を駆動する基になるルーチンへのアクセスを提供するプログラミングインターフェイスです。 アドイン、ウィザード、およびマクロは、オートメーションモデルのオブジェクトを使用して、IDE の機能を制御または拡張します。  
  
 コマンド UI コンテキスト  
 UI コマンドまたは要素 (ツールバーなど) の可視性と GUID の関連付け。 コマンド UI コンテキストは、ウィンドウにアタッチされていないので、選択コンテキストとは異なります。  
  
 コマンド UI コンテキストは、次の場合に使用できます。  
  
- 特定のウィンドウがアクティブになったときに表示されるツールバーに GUID を割り当てます。  
  
- VSPackage を読み込んだり実行したりすることなく、コマンドの可用性に GUID を割り当てます。  
  
- アクティブなキーバインドに影響を与える GUID を割り当てます。  
  
- マクロの記録を有効にする GUID を割り当てます。  
  
- デバッグモードをアクティブ化したり、エディターでデザインモードと実行モードを切り替えるために、GUID を割り当てます。  
  
  コンポーネント  
  アプリケーションの機能の一部として、ソフトウェアの実装に関する既存の情報が含まれていないソフトウェア。 コンポーネントとアプリケーションの間の通信は、OLE スタイルのインターフェイスを介してのみ行われます。  
  
  コンポーネントマネージャー  
  `SOleComponentManager`最上位レベルのコンポーネントに対して非ユーザーインターフェイスの調整サービスを提供するサービス。 この `SOleComponentManager` サービスは、インターフェイスを実装し `IOleComponentManager` ます。  
  
  コンポーネント UI マネージャー  
  `SOleComponentUIManager`ユーザーインターフェイス調整サービスを提供するサービス。 サービスは、 `SOleComponentUIManager` `IOleComponentUIManager` インターフェイスとインターフェイスを実装し `IOleInPlaceComponentUIManager` ます。  
  
  コンテキストバッグ  
  `IVsUserContext`環境コンポーネントにアタッチされたオブジェクト (COM オブジェクト)。 このオブジェクトは、検索キーワード、F1 キーワード、およびコンポーネントに関連する属性を保持します。 また、コンテキストバッグは、リンクされている任意の subcontext バッグを指します。  
  
  コンテキストプロバイダー  
  コンテキストバッグが関連付けられている IDE 内のコンポーネント。 このようなコンポーネントには、ツールウィンドウ、エディター、またはプロジェクトの階層が含まれます。  
  
  designer  
  UI (フォーム、ボタン、およびその他のコントロール) の要素をユーザーが操作できるようにするプログラミングインターフェイス。  
  
  DocData  
  ドキュメント/ビューの分離が存在する、ドキュメントの基になるデータをカプセル化する COM オブジェクト (たとえば、テキストエディターの場合は、すべてのテキストエディタービューの基になるテキストバッファーになります)。 EditorFactory がこのオブジェクトを提供していない場合、IDE はその代わりに1つを製造します。 このオブジェクトの役割は、データの永続性を管理し、複数のビューの共有セマンティクスを同じものにすることです `DocData` 。 オブジェクトがインターフェイスをサポートしている場合は、 `DocData` `IOleCommandTarget` uishell のコマンドルーティングに含まれます。  
  
  DocObject  
  ホストによって提供されるフレーム内で UI をホストするために使用されるテクノロジ。 具体的には、この用語はと関連するインターフェイスをサポートする埋め込みを指し `IOleDocument` ます。 このテクノロジには、COM ドキュメントの実装の詳細、Visual Basic 5.0 のツールウィンドウ、Visual Basic 6.0 の ActiveX デザイナーなど、さまざまなアプリケーションが含まれています。  
  
  ドキュメント  
  との両方をドキュメント全体として参照するために使用され `DocData` `DocView` ます。 たとえば、DocumentFrame にはが含まれてい `DocView` ますが、への参照を保持 `DocData` して永続化を処理することもできます。  
  
  DocView  
  基になるを表示および操作するためにユーザーがやり取りする、DocObject/埋め込み/ウィンドウペイン `DocData` 。 ユーザーは、インターフェイスデザインの一部であるドキュメント/ビューの分離を利用していないことに注意してください `DocObject` 。 ユーザーは、として知られている基になるデータの抽象的な概念 (および形式化されていない) を使用するのではなく、DocObject 全体を使用してビューとして機能し `DocData` ます。 `DocView` オブジェクトは、常に IDE のドキュメントフレームオブジェクト (MDI 子ウィンドウ) と共に埋め込まれます。  
  
  DTE  
  `DTE`(開発ツールの機能拡張) オブジェクトは、Visual Studio オートメーションモデルの最上位のアクセスポイントです。このオブジェクトを使用して、IDE をプログラムで自動化したり拡張したりできます。  
  
  [ダイナミック ヘルプ] ウィンドウ  
  IDE によって実装されているツールウィンドウで、lookup キーワードまたは F1 ヘルプトピックの一覧を表示します。  
  
  エディター  
  を実装するコード (クラス、CLSID) `DocView` 。 また `DocData` 、ビューとデータの分離がサポートされている場合にもを実装します。  
  
  拡張機能  
  IDE に対して変更、カスタマイズ、または追加を行う機能。 拡張機能は、オートメーションモデルまたは Vspackage を使用して作成します。  
  
  外部エディター  
  Microsoft Word など、IDE に固有ではないエディター。 独自のメカニズムを使用して登録されており、IDE の外部で使用できます。 このエディターを埋め込むことができる場合は、IDE のウィンドウ内に表示されます。 埋め込むことができない場合は、別のトップレベルウィンドウが作成されます。  
  
  階層  
  ノードのツリー。各ノードは、一連のプロパティに関連付けられています。  
  
  独立した最上位レベルのコンポーネント  
  モードレストップレベルウィンドウを使用するコンポーネントで、スタンドアロンのアプリケーションウィンドウとして効果的に動作できますが、インプロセスオブジェクトとして実装されています。 そのため、独立した最上位のコンポーネントでは、IDE でモダリティとメッセージループサービスを調整する必要があります。 インプロセスオブジェクトには、独自のメッセージループはありません。  
  
  情報プロバイダー  
  情報プロバイダーは、キーワードを検索し、トピックのリストをオブジェクトの形式で返すことができるモジュールです `IVsUserContextItem` 。 情報プロバイダーに F1 キーおよび lookup キーワード項目を提供するには、コンパイル済みヘルプファイル () を登録します。HxS) を使用します。 これらのファイルのヘルプトピックは、[ダイナミックヘルプ] ウィンドウに表示されるトピックの一覧を提供し、ユーザーが F1 キーを押したかどうかを示すために使用されます。  
  
  インプレースコンポーネント  
  `IOleInPlaceComponent`IDE によって所有されるドキュメントウィンドウ内に視覚的に含まれるウィンドウを管理するためのインターフェイスを実装する VSPackage オブジェクト。 インプレースコンポーネントは、標準の OLE メニューには参加しません。代わりに、ユーザーインターフェイス要素を IDE に統合します。  
  
  インプレースコンポーネントには、固定された埋め込みコンポーネントとコンポーネントコントロールの2種類があります。  
  
  固定されたインプレースコンポーネントには、ide に直接組み込まれているかのように表示される、IDE のユーザーインターフェイスに緊密に統合されたメニュー、ツールバー、およびコマンドがあります。  
  
  コンポーネントコントロールには、IDE に統合された独自のユーザーインターフェイス要素はありません。代わりに、IDE のメニュー、コマンド、およびツールバーを使用します。 たとえば、太字のコマンドを使用して、フォームに埋め込まれているリッチテキストコントロール内の選択された単語を太字にすることができます。 ただし、コンポーネントコントロールは、動的にインストールされたコンポーネント固有の UI 要素を表示するように要求できます。  
  
  言語サービス  
  VSPackage 開発者が、テキストマークや色分けなどのコンピューター言語コードエディターの機能を実装できるようにするオブジェクトのセット。  
  
  その他のファイルプロジェクト  
  プロジェクトは、プロジェクトに含まれていない開いているファイルを格納するために使用されます。 このプロジェクト内の項目の一覧は保存されません。  
  
  プロジェクト  
  プロジェクトは、階層オブジェクト、またはインターフェイスを実装する COM オブジェクトで構成され `IVsHierarchy` ます。  
  
  プロジェクト固有のデザイナーまたはエディター  
  プロジェクトの種類とは無関係に使用できないデザイナー。 プロジェクト固有のすべてのデザイナーは、エディターのファクトリ情報をレジストリに入力する必要があります。 特定のプロジェクトで特定のファイルの種類が開かれるたびに、IDE がデザイナーをインスタンス化できるようになります。  
  
  プロジェクトの種類のウィンドウ  
  グローバル選択コンテキストから、現在アクティブなプロジェクトの階層と項目を常に追跡するウィンドウ。 プロジェクトタイプのウィンドウサービスを使用して、 `SVsTrackSelectionEx` 変更の IDE にアラートを通知し、ユーザーにフィードバックを表示します。 ソリューションエクスプローラーは、プロジェクトの種類のウィンドウの例です。  
  
  [プロパティ] ウィンドウ  
  以前のプロパティブラウザー。  
  
  参照ベースのプロジェクト  
  プロジェクトのファイルを同じディレクトリに配置する必要がないプロジェクト。 代わりに、他の関連のないディレクトリからのファイルへの参照は、プロジェクト自体によって格納および管理されます。  
  
  実行 (ドキュメントテーブルを)  
  現在開いているすべてのドキュメントの一覧を IDE で保持する内部構造。 この一覧には、ドキュメントが現在編集されているかどうかに関係なく、メモリ内のすべての開いているドキュメントが含まれます。 ドキュメントとは、エディターで開かれたストアドプロシージャ、プロジェクト内のファイル、またはメインプロジェクトファイル (* .vcproj ファイルなど) を含む、保存されているアイテムのことです。  
  
  選択コンテキスト  
  IDE 内のすべてのウィンドウの詳細に含まれるデータ。アクティブな選択を追跡するために使用されます。 選択コンテキストは次のもので構成されます。  
  
- `IVsHierarchy`プロジェクト階層のインターフェイスへのポインター  
  
- プロジェクト項目の項目識別子。  
  
- `ISelectionContainer`アクティブなオブジェクトのプロパティへのアクセスを提供するインターフェイスへのポインター。  
  
- 要素の値の配列。  
  
  サービス (service)  
  1つの COM オブジェクト内に存在する一連の COM インターフェイスのコントラクト。 GUID によって識別されるサービスを作成する場合は、サービスを実行する COM インターフェイスのセットを定義します。 COM オブジェクトは、サービスを使用して相互に通信します。  
  
  ソリューション  
  ユーザーが動作する関連プロジェクトのグループ。  
  
  標準デザイナー  
  プロジェクトの種類に関係なく使用できるデザイナー。 すべての標準デザイナーは、エディターのファクトリ情報をレジストリに入力する必要があります。 IDE は、特定の拡張子を持つファイルを開くたびに、デザイナーをインスタンス化できます。 データはファイルに保存する必要があります。  
  
  標準エディター  
  特定の種類のプロジェクトに依存せずに使用できるエディター。 このようなエディターでは、EditorFactories がレジストリに登録されています。 これにより、IDE がエディターを見つけて起動できるようになります。  
  
  標準 OS エディター  
  Visual Studio 固有ではない埋め込み。 既知の Win32 キーを使用して登録されます (たとえば、Win32 Explorer がを呼び出す方法を認識している場合)。 このようなエディターを埋め込むことができる場合でも、エディターは IDE の代わりに表示されます。 それ以外の場合は、このようなエディター用に別のトップレベルウィンドウが作成されます。  
  
  subcontext バッグ  
  `IVsUserContext`コンテキストバッグにリンクされているオブジェクト。 このオブジェクトは、IDE コンポーネント内の選択項目の参照キーワード、F1 キーワード、および属性を保持します。 Subcontext の例としては、ツールウィンドウにコマンドが含まれている場合や、エディターにキーワードが含まれている場合があります。  
  
  タスク一覧  
  IDE によって実装されるツールウィンドウで、アクティブなタスクの一覧が表示されます。  
  
  テキストバッファー  
  オブジェクトの共通名 `VSTextBuffer` 。  
  
  テキストビュー  
  オブジェクトの共通名 `VSTextView` 。  
  
  ツールトップレベルコンポーネント  
  モードレスのポップアップウィンドウとして動作し、IDE のユーザーインターフェイスと緊密に連携するコンポーネント。 独立した最上位レベルのコンポーネントと同様に、ツールの最上位コンポーネントでは、モダリティとメッセージループサービスを IDE で調整する必要もあります。  
  
  最上位のコンポーネント  
  IDE ウィンドウのクライアント領域ではなく、モードレスのトップレベルウィンドウを管理する VSPackage オブジェクト。 最上位のコンポーネントには、 `IOleComponent` アイドル時間へのアクセスなどのメッセージループサービスを利用するためのインターフェイスが実装されています。  
  
  アクティブな UI  
  現在フォーカスがある、表示されている VSPackage オブジェクト。  
  
  UI 階層  
  `IVsUIHierarchy`階層の表示を可能にするインターフェイスを実装する COM オブジェクト。 UI 階層ウィンドウは、 `ISelectionContainer` プロパティウィンドウを更新するためのインターフェイスを実装します。他のプロジェクトタイプのウィンドウは、必要に応じて、この実装を使用できます。  
  
  .VSCT  
  Visual Studio コマンドテーブル。 Vsct ファイルには、XML 形式のメニュー、ツールバー、およびコマンドの配置と動作に関する情報が含まれています。  
  
  VSPackage  
  Visual Studio IDE を拡張するインストール可能なソフトウェア。ユーザーインターフェイス、サービス、プロジェクトの種類、エディター/デザイナーの1つ以上の情報を提供します。 VSPackage は、インターフェイスを実装する COM オブジェクト `IVsPackage` と、選択やその他の機能をサポートする他のインターフェイスを実装するその他の com オブジェクトで構成されます。 また、VSPackage には特定の登録要件があります。
